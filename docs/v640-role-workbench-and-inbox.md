# V6.4.0 四角色工作台与统一收件箱实施说明

日期：2026-07-15  
版本：6.4.0  
状态：功能、生产数据库迁移和自动化验收已完成；四个真实人员的受控试点留到 V6.6.0

## 1. 本版本解决什么问题

V6.4.0 不再把 PM、运营、PMO 和 CEO 当作同一个用户展示同一套首页。系统继续共用一套业务数据底座，但根据当前业务角色呈现不同工作重点：

| 角色 | 首屏目标 | 关键内容 |
|---|---|---|
| PM | 今天应该推动什么 | 我的项目、关键路径、里程碑、重大风险、交付物、今日行动、正式周报 |
| 运营 | 哪些经营动作正在阻塞 | 验收、开票、应收、回款、现金、责任人、期限与关闭证据 |
| PMO | 哪些例外需要治理 | 红黄灯、风险、资源冲突、数据质量、治理 SLA、会议与升级事项 |
| CEO | 现在必须做什么决策 | 战略项目、现金、收益、重大风险、待决策事项、决策回执与效果 |

统一收件箱聚合风险、联合检查、运营日历、治理审批、管理信号、AI 建议、决策回执、飞书确认和正式业务成果。每条事项都保留项目、来源、更新时间、数据分类和业务入口，不以通知列表替代正式业务状态。

## 2. 用户输入真实化

以下页面已改为业务选择器和结构化字段编辑，不再要求业务用户手填 UUID 或 JSON：

- 运营闭环：项目、责任人、复核人、业务对象和证据选择器。
- 复盘知识：证据、责任人和来源对象选择器。
- 项目影响包：源对象、目标对象、责任人、证据以及结构化变化/影响字段。
- 项目生命周期：业务对象、责任人、证据、目标状态和结构化纠偏内容。
- 决策中心：结构化汇报指标、决策输入、证据和行动关闭证据。
- PMO 控制中心：治理证据选择器。
- 黄金链验收：正式成果和证据选择器，参与人显示业务名称。
- 知识引用审计：选择正式业务成果，不输入成果 ID。
- 风险通知：只使用当前用户已配置的个人飞书连接；缺少配置时明确引导到用户中心，不静默借用管理员身份。

`GET /api/business-directory` 是上述选择器的统一业务目录，返回项目、人员、证据、正式成果和项目业务对象的显示名称、稳定 ID、项目范围、状态和来源信息。内部仍以稳定 ID 关联，界面只展示业务可理解的名称。

## 3. 关键接口

### `GET /api/role-workbench`

强制校验 `org_id、subject_scope、subject_id、business_role、data_class`，返回角色焦点、工作区块、来源、更新时间、警告和 CEO 经营摘要。不同角色只读取授权范围内的项目与业务事实。

### `GET /api/business-directory`

支持 `project_id` 与 `entity_type` 过滤，返回：

- `projects`
- `people`
- `evidence`
- `formal_outputs`
- `business_objects`

### `GET/POST /api/collaboration-inbox`

读取统一收件箱并持久化当前用户对事项的查看、确认、延后和关闭回执。回执不修改来源业务对象，只记录协同处理状态；业务完成仍必须回到来源模块执行。

## 5. 四角色自动化验收空间

为避免继续使用同一用户自循环，系统建立了独立 `test` 数据空间：

- 一个隔离测试组织；
- PM、运营、PMO、CEO 四个不同测试账号；
- 五个测试项目；
- PM → PMO、运营 → PMO、PMO → CEO 的真实汇报与升级关系；
- 验收运行、参与人、隔离结果和证据持久化。

账号凭据仅保存在本机被 Git 忽略且权限为 `0600` 的临时文件，不写入代码、数据库说明、知识库、日志或发布产物。自动化账号用于验证认证、权限、角色隔离和页面合同，不等同于四个真实人员的试点签字。

## 6. 验收门槛

- 四个业务角色映射到四个不同账号，不允许自循环。
- 五个测试项目全部处于 `test` 空间，不进入 `production`。
- 四角色工作台内容和重点不同。
- 所有目录、工作台和收件箱请求执行组织、项目、角色和数据分类校验。
- 正式页面不存在要求用户输入 UUID、JSON、记录 ID 或用户 ID 的业务表单。
- 个人飞书动作不静默回退到管理员连接。
- 迁移对象、RLS、客户端授权、函数和触发器通过生产审计。
- 四个真实人员的业务试点、两条黄金链和三类飞书真实写入在 V6.6.0 完成，未达到时不得宣布 V7.0。
