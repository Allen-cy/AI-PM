# V6.6.7 数据分类目标镜像自动闭环

## 目标

V6.6.6 已经可以把分类决定持久化，并在个人飞书、高风险二次确认、当前值复核、同步流水和租约保护下修改飞书中文字段“数据分类”。当时写回成功后仍需人工执行一次全量对账，确认事项也可能在稳定镜像形成前关闭。

V6.6.7 将最后一段改为系统内自动闭环：

1. 校验个人飞书连接与组织共享事实源为同一 Base、同一领域表和同一同步流水表。
2. 在个人身份下写入飞书“数据分类”，并完成同步流水。
3. 由组织事实源立即读取同一飞书记录。
4. 以目标 `data_class` 执行单记录、非完整快照对账。
5. 只有目标镜像完成且失败数、隔离数均为 0，才关闭确认事项。

## 数据与安全契约

- 定向范围必须包含 1 至 100 个明确的飞书记录 ID；本闭环固定为 1 个。
- 完整快照不能混入定向记录范围，定向请求不能缺少记录 ID。
- 飞书实际返回的记录 ID 必须与申请完全一致，否则以 `TARGET_RECORD_SCOPE_MISMATCH` 拒绝。
- 定向对账调用 `apply_feishu_reconcile_domain_tx` 时固定传入 `p_full_snapshot=false`，不会把未扫描到的其他记录标记为墓碑。
- 请求指纹包含快照类型和记录 ID；幂等键同时包含分类草稿版本和写回尝试次数。
- 个人飞书不得静默回退为管理员或组织身份；个人和组织台账不一致时以 `V667_PERSONAL_ORGANIZATION_BASE_MISMATCH` 阻断。

## 失败与重试

飞书已写入但目标镜像失败时，同步流水不被误标成飞书写入失败，确认事项保持失败且可重试。再次执行会先读取飞书当前值：若目标分类已存在，则跳过外部更新，只重新运行目标镜像。这样既保留真实外部事实，也避免重复写入。

## 数据库影响

本版本不新增 migration。实现复用 V6.2 已上线的：

- `begin_feishu_reconcile_batch_tx`
- `apply_feishu_reconcile_domain_tx`
- `finalize_feishu_reconcile_batch_tx`
- `fail_feishu_reconcile_batch_tx`

V6.6.7 仅启用既有 RPC 的非完整快照能力，不重跑历史 SQL。

## 验收边界

- 自动化测试覆盖同 Base 门禁、单记录目标镜像、非完整快照和越界记录拒绝。
- 数据库事务烟测只能使用合成数据并完整回滚。
- 发布验收不执行真实飞书写回，不替业务负责人分类现有 237 条隔离记录。
- 五个真实项目、四位不同真实人员、黄金链 A/E 和三类真实飞书回执继续是 V7.0 人工门禁。

## 生产配置核验

- 线上 `/api/integrations/feishu/health` 使用一次性测试账号只读验收，返回组织级来源、Base 可访问、8/8 表映射完整、缺失表 0；账号和会话随后删除。
- Supabase 当前 `organization_feishu_connections=0`，组织共享读取由 Vercel 运行时敏感环境变量提供；`vercel env pull` 对敏感值返回空占位，不能据此判断线上未配置。
- Supabase 当前 `user_feishu_connections=0`，因此个人写回按设计保持不可执行；系统不得把组织共享应用静默当成个人身份。
- 本机 `lark-cli` 只读确认“AI-PMO业务管理底座”及项目、里程碑、任务、风险、合同、回款、成本、同步账本 8 张表真实可访问；未执行新增、修改或删除记录。
