# AI-PMO 后续优化计划（2026-07-03）

## 一、当前系统位置

AI-PMO 已经从早期“功能集合”推进到 PM/PMO 日常管理闭环的骨架阶段：项目组合看板、工作台、治理工作流、风险管理、报告工厂、权限审计、飞书接入、AI依据审计、迁移中心等核心模块已具备可运行入口。

当前最需要继续增强的不是再增加孤立页面，而是把关键管理动作做成“输入清楚、处理过程可追溯、输出能复核、责任能闭环”的事务链。

## 二、后续优化主线

### 主线 A：迁移与数据接入从评估走向落地

目标：让竞品A或 Excel/飞书历史数据能够经过评估、试迁移、整改、复核、正式切换，形成可审计的迁移过程。

任务：

- P12-T5：迁移质量问题自动生成整改行动项。（已在 V5.3.15 落地）
- P12-T6：迁移整改行动项持久化到 Supabase，支持责任人、截止日期、状态流转。（已在 V5.3.16 落地）
- P12-T7：迁移整改项飞书任务回写确认队列。（已在 V5.3.17 落地）
- P12-T8：字段映射方案保存与复用，避免每次上传都重新判断字段口径。（已在 V5.3.18 落地）
- P12-T9：试迁移批次对比，比较多轮试迁移的字段覆盖率和问题关闭率。（已在 V5.3.19 落地）
- P12-T10：正式迁移前检查清单，输出 Go/No-Go 迁移决策材料。（已在 V5.3.20 落地）

验收标准：

- 用户上传样本后，不只看到问题，还能看到每个问题对应的整改任务。
- 每个整改任务有责任角色、建议截止日期、优先级、验收标准。
- 后续版本能把整改任务保存、流转、关闭，并体现在迁移评审报告中。

### 主线 B：PMO 治理从流程展示走向可运营

目标：治理中心不只是展示流程，而是能处理立项、阶段门、风险升级、变更、收尾验收等真实审批动作。

任务：

- P13-T1：治理流程实例增加 SLA、逾期提醒、待我处理视图。（已在 V5.3.21 落地）
- P13-T2：治理审批结果与项目状态、风险状态、报告工厂联动。（已在 V5.3.22 落地）
- P13-T3：治理流程导出审计包，包含输入、审批意见、附件、输出成果。（已在 V5.3.23 落地）
- P13-T4：治理策略配置化，支持不同项目等级使用不同阶段门。（已在 V5.3.24 落地）

验收标准：

- PM/PMO 能从工作台进入待办治理事项。
- 每个治理流程有状态、责任人、审批人、期限、输出物和审计记录。

### 主线 C：风险管理继续升级为事务闭环

目标：风险管理模块从“登记/分析/跟踪”升级到“识别-分析-应对-实施-监督-关闭”的完整事务系统。

任务：

- P14-T1：风险与项目台账、任务、里程碑、回款联动。（已在 V5.3.25 落地）
- P14-T2：风险升级自动触发治理流程或行动项。（已在 V5.3.26 落地）
- P14-T3：风险敏感性分析结果进入报告工厂和项目健康矩阵。（已在 V5.3.27 落地）
- P14-T4：风险关闭必须提交关闭证据和复核意见。（已在 V5.3.28 落地）

验收标准：

- 高风险不能停留在只读报告中，必须有责任人、应对动作、deadline 和关闭证据。
- 风险变化能影响项目健康、工作台提醒和报告内容。

### 主线 D：飞书与个人配置从系统级接入走向用户级接入

目标：不同用户使用自己的飞书连接、AI模型配置和项目访问范围。

任务：

- P15-T1：用户级飞书配置完整诊断和引导。（第一版已在 V5.3.48 落地：用户中心支持一键测试个人飞书 App、Base Token、表 ID、字段权限和可选写入权限；后续继续把逐项修复指引嵌入更多业务页）
- P15-T2：飞书连接状态在相关功能页内实时提示。（第一版已在 V5.3.48 落地：集成中心新增统一集成状态组件，展示当前账号 AI、飞书、RAG、同步审计状态；后续继续覆盖项目看板、风险、治理、报告等业务页）
- P15-T3：飞书写入动作统一进入待确认队列，避免误写业务表。（部分完成：迁移整改、知识治理待办、知识治理周提醒、治理流程等关键写入已要求 prepare/confirm 或 confirmation_required；待把通用 `/api/integrations/feishu/actions` 也纳入用户级确认队列和可视化待办）
- P15-T4：用户级 AI 模型配置增加可用性测试和失败降级说明。（第一版已在 V5.3.48 落地：新增用户模型连通性测试 API、用户中心测试按钮、失败原因分级和下一步建议；后续可补测试历史和批量模型比较）

验收标准：

- 用户使用飞书相关功能时，系统能明确告诉用户当前用的是谁的飞书、缺什么配置、下一步怎么补。
- 写入飞书前有明确确认、记录和失败补偿。

### 主线 E：知识库与 RAG 从问答走向可追溯知识运营

目标：RAG 不只是回答问题，而是能说明引用来源、版本、适用场景和更新责任人。

任务：

- P16-T1：知识条目增加生命周期：草稿、已评审、已发布、过期。（部分完成：RAG 类型已支持 `draft/reviewed/published/deprecated/archived`，风险复盘资产支持发布、归档和治理动作；待做独立知识条目管理页、过期提醒和版本流转）
- P16-T2：报告、AI建议、治理结论均能引用知识库来源。（部分完成：RAG 问答有引用，AI依据审计和报告工厂已有来源结构，风险复盘资产可进入动态 RAG；待把治理结论、流程输出和 AI 建议统一绑定知识引用）
- P16-T3：知识库更新后自动形成变更摘要和影响模块。（未完成：尚未实现知识变更 diff、影响模块计算、订阅提醒和变更审计流）
- P16-T4：PMO制度、模板和最佳实践形成可检索目录。（部分完成：静态 corpus、模板中心和风险复盘资产已可检索/下载；待做 PMO 制度、模板、最佳实践的统一目录、筛选和生命周期管理）

验收标准：

- AI输出必须能解释依据，不确定或缺数据时拒答。
- 知识库更新能反向提醒系统模块和使用者。

### 三-B、跨设备继续开发建议

如果在另一台设备继续开发，建议优先同步 GitHub 最新 `main`，确认 tag `v5.3.48` 存在，然后按以下顺序继续：

1. 先执行本仓库已有数据库脚本，尤其是 `supabase-v529-governance-workflows.sql`、`supabase-v530-issue-change-action-chain.sql`、`supabase-v5338-risk-retrospective-governance-followups.sql`、`supabase-v5344-risk-retrospective-governance-operations.sql`、`supabase-v5347-knowledge-governance-evidence-chain.sql`。
2. 先跑 `npm test` 和 `npm run build`，确认另一台设备环境一致。
3. 继续开发 V5.3.49 时优先补通用飞书写入确认队列、可视化确认待办和全页面统一状态组件覆盖，不要先新增孤立展示页。
4. 任何飞书写入和业务主数据反写都必须保持“先预览/候选/待确认，用户确认后再写入”的边界。
5. 新增用户可见功能后必须按全局发布规则升版本、构建、打 tag、创建 Release，并确认 Vercel Production 部署成功。


## 四、V5.3.15 执行结果

主题：迁移质量问题自动生成整改行动项。

已完成范围：

- 从试迁移分析结果中生成结构化整改行动项。
- 每个行动项包含：标题、优先级、责任角色、建议截止日期、来源问题、样例、验收标准、建议动作。
- 迁移中心页面新增“整改行动项”区域。
- 迁移评审报告同步纳入整改行动项。

不做：

- 暂不写入 Supabase。
- 暂不写入飞书任务。
- 暂不做多人审批流。

验收标准：

- 质量问题能转化为可执行任务，而不是只显示问题描述。
- 高优先级问题对应 P0 行动项。
- 无质量问题时仍给出试迁移复核行动项。
- 测试、构建、发布和知识库同步全部完成。

## 五、V5.3.16 执行结果

主题：迁移整改行动项持久化与状态流转。

已完成范围：

- 新增 Supabase 脚本保存迁移整改行动项。
- 支持行动项状态：待处理、处理中、待复检、已关闭。

## 六、V5.3.17 执行结果

主题：迁移整改项飞书任务回写确认队列。

已完成范围：

- 为已保存的迁移整改行动项增加“准备同步飞书任务”的确认入口。
- 新增待确认队列：未同步、待确认、同步中、已同步、同步失败。
- 写入飞书前要求用户显式确认，避免误写外部任务。
- 成功写入后回填飞书任务 GUID 和链接；失败时记录错误原因和请求 ID。
- 复用个人飞书配置优先、全局飞书兜底的现有飞书任务能力。
- 每次准备同步和确认写入都写入操作审计。

验收标准：

- 不自动写飞书；用户必须显式确认。
- 每次写入都有审计记录和失败提示。
- 系统内整改项仍是主记录，飞书任务只是协同执行通道。

边界：

- 当前版本不做飞书任务责任人自动匹配 open_id，任务说明中保留责任角色和责任人文本；后续可在用户中心或飞书配置中增加“系统用户-飞书 open_id”映射后再自动指派。
- 需要先在 Supabase 执行 `supabase-v5317-migration-remediation-feishu-sync.sql`，否则页面会提示缺少飞书同步字段。

## 七、V5.3.18 执行结果

主题：字段映射方案保存与复用。

已完成范围：

- 保存每次试迁移的字段映射方案，包括源字段、目标中文字段、匹配方式和人工确认结果。
- 支持按数据对象复用历史映射，避免同类迁移包每次重新判断字段口径。
- 对字段变化做差异提示：历史来源字段缺失、当前来源字段新增、目标字段映射变化、当前仍未映射字段。
- 新增 `supabase-v5318-migration-field-mapping-profiles.sql`，字段映射方案可持久化到 Supabase。
- 新增 `/api/migration/field-mappings`，支持按数据对象读取字段映射方案和保存当前分析结果。
- `/migration-center` 新增“保存字段映射方案”和“字段映射方案库”；选择历史方案后先显示匹配度、差异项和风险提示，不会静默套用。

验收标准：
