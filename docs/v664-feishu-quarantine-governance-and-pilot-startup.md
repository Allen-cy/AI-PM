# V6.6.4 飞书隔离分类治理与正式试点启动包

版本：6.6.4

日期：2026-07-16

状态：技术实现完成；正式受控试点仍等待真实项目与真人证据

数据库迁移：无新增迁移，复用 `feishu_reconcile_quarantine` 和 V6.6 试点门禁

## 1. 问题与结论

V6.2 已安全隔离首次飞书对账中的 237 条记录，但 V6.6.3 页面只能显示隔离数量，没有逐条治理入口。生产只读审计进一步确认：

- 唯一隔离记录 237 条：项目 60、合同 56、回款 62、风险 56、同步账本 3。
- 4 次对账累计读取 948 条、隔离 948 条、失败 0；同一 237 条记录被幂等重复识别，并非 948 条不同业务记录。
- 原始载荷启发式扫描在234条记录中检出“样例来源”或“测试批次”等字样；这不是最终分类结果。
- 生产治理API只使用白名单业务证据，最终输出为230条样例建议、7条人工判断、0条正式候选；后续治理与CSV以此为准。
- 当前 production 项目镜像 0，正式项目候选 0；test 项目及稳定映射各 5 条。
- 数据库只有 2 个真实用户账号，四角色仍由同一真实用户承担，最大职责分离匹配为 1/4。

因此不能把现有样例数据批量改成 production 来凑齐五项目。V6.6.4 的目标是提供可执行的治理工具，同时继续保护正式试点真实性。

## 2. 页面与使用路径

### 2.1 飞书隔离数据分类治理台

路径：`/integration-center/data-governance`

仅允许已授权的组织级 PMO 访问。页面展示：

- 待治理总数、建议样例数、待人工判断数和明确正式候选数。
- 各领域记录数及分类建议分布。
- 记录名称、项目编号、隔离原因、建议分类、建议依据、飞书需填写的中文字段和值。
- UTF-8 中文 CSV 下载入口。
- 返回集成中心重新执行八领域对账的路径。

接口不会把 `source_payload` 原始整行返回给浏览器，只返回分类治理所需的白名单摘要。

### 2.2 正式试点启动包

路径：`/operations-center/pilot-acceptance`

组织级 PMO 可点击“下载正式试点启动包”，导出：

- 五个 production 项目的当前数量与候选清单。
- PM、运营、PMO、CEO 四角色真人最大匹配与候选清单。
- 黄金链 A/E 和飞书消息、任务、智能表写入的准备度。
- 隔离数据的领域与分类建议汇总。
- 推荐执行顺序和不可自动代办的真人证据。

## 3. 分类建议规则

1. 飞书明确填写中文字段“数据分类/数据空间/数据类型”为正式、样例、测试或诊断时，按显式值建议。
2. 含“样例来源/样例编号”等标记时，只建议进入样例空间。
3. 无样例标记但含“测试批次/测试标记”时，只建议进入测试空间。
4. 含诊断标记时建议进入诊断空间。
5. 缺少上述证据时保持“待人工判断”，绝不自动推断为正式。
6. 只有项目记录且显式分类为正式时，才计入正式项目候选。

建议只用于辅助核对，不直接修改飞书记录，也不修改 Supabase 隔离状态。业务负责人在飞书补齐中文字段后，重新对账会按原有 V6.2 原子流程写入正确空间，并自动关闭已经通过门禁的隔离记录。

## 4. API 契约

### 4.1 读取分类治理清单

`GET /api/integrations/feishu/quarantine-governance`

必填查询参数：

- `role=pmo`
- `org_id`
- `subject_scope=organization`
- `subject_id=<org_id>`
- `data_class`

响应遵循统一契约：`status、request_id、context、source、data_class、generated_at、warnings、data`。`data` 包含总数、汇总和最多 1000 条白名单化建议。接口按 `org_id + data_class + pending/under_review` 强制过滤；没有 POST，避免绕过飞书源端确认。

### 4.2 下载中文 CSV

`GET /api/integrations/feishu/quarantine-governance?...&format=csv`

CSV 以 UTF-8 BOM 输出，包含隔离记录 ID、飞书记录 ID、领域、名称、项目编号、原因、建议分类、依据、置信度、必填中文字段和值。

### 4.3 下载正式试点启动包

`GET /api/operations-center/pilot-acceptance?...&format=startup-pack`

只允许组织级 PMO。输出 Markdown，不要求先创建试点批次；它是启动检查与操作清单，不是正式验收报告。

## 5. 安全与边界

- 不删除、不覆盖、不自动重分类 237 条历史记录。
- 不把样例或测试数据提升为 production。
- 不返回飞书凭据、Supabase 密钥、用户密码或完整原始载荷。
- 不代替四位真实人员签署。
- 不自动执行飞书外部写入；原有人工确认队列保持不变。
- 不把启动包视为验收证据；正式通过仍由数据库根据真实事实计算。
