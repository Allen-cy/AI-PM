# V6.6.2 四角色汇报关系职责分离门禁

版本：6.6.2

数据库迁移：`20260716004254_v662_reporting_separation_guard.sql`

## 1. 审计发现

V6.4 已为技术演练建立 PM、运营、PMO、CEO 四个不同测试账号及正确的上行关系，但旧组织仍有 3 条活动关系由同一用户跨角色向自己汇报或升级。它们不会满足正式试点的“四位不同真实人员”要求，也不应继续参与治理路由。

## 2. 处理原则

- 不删除旧关系，不覆盖历史创建时间和参与角色。
- 将旧自循环关系改为 `suspended`，补充暂停原因和有效期终点。
- 新增已验证的数据库约束：`from_user_id = to_user_id` 时，状态不能是 `active`。
- 管理员接口在写数据库前直接拒绝同一用户自循环。
- 管理员页面过滤当前上报人，并在选择冲突时禁用提交。

## 4. 正式试点边界

技术演练仍由四个不同测试账号承担，5 个测试项目和 16/16 模块证据不受影响。正式试点仍需要四位不同真实人员、至少 5 个 production 项目、本人签署、黄金链 A/E、飞书三类成功回执和一次真实失败恢复；本迁移不会自动生成这些业务证据。
