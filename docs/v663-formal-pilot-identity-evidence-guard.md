# V6.6.3 正式试点身份与证据门禁

## 1. 问题与目标

V6.6.0–V6.6.2 已建立受控试点、四角色职责分离和正式通过硬门槛，但正式试点仍有两处需要在数据库层闭合：参与人的“真实/测试”属性不能只依赖接口提交，飞书成功回执也必须具备稳定项目范围才能成为验收证据。

V6.6.3 的目标是让账号身份和外部证据资格由数据库独立判断，并在正式试点页面给出可执行的启动检查。系统继续保留所有历史记录，但不让不合格记录进入正式验收。

## 2. 用户体验

PMO 打开 `/operations-center/pilot-acceptance` 并选择 production 数据空间后，可看到四项启动检查：

1. 至少五个 production 项目；
2. PM、运营、PMO、CEO 可由四位不同真实用户匹配；
3. 黄金链 A 和 E 已正式通过；
4. 飞书消息、任务、智能表更新三类成功回执均有稳定项目范围。

每项检查提供直接补齐入口。测试账号不会出现在正式试点候选中，无项目范围的历史飞书回执不会出现在证据候选中。技术演练继续只使用隔离测试账号和 test 数据。

## 3. 数据库契约

迁移：`20260716123000_v663_formal_pilot_identity_evidence_guard.sql`、`20260716124000_v663_account_kind_change_guard.sql`

生产预发布审计曾登记较早的 `20260716012841` 时间戳。该编号保留为无副作用兼容标记，避免修改或删除迁移历史；真正的身份与证据门禁固定在 V6.6.0 基础迁移之后执行，保证新环境从零按文件名顺序部署也能成功。

- `app_users.account_kind`：`real_user | test_account | service_account`，默认 `real_user`。
- 已参与技术演练并标记为测试参与人的历史账号回填为 `test_account`。
- `controlled_pilot_participants(run_id,user_id)` 唯一，禁止同一账号在同一批次承担两个角色。
- `enforce_v663_pilot_participant_identity()`：正式试点必须使用真实用户，技术演练必须使用测试账号。
- `enforce_v663_pilot_feishu_project_scope()`：试点飞书证据必须来自成功确认，且项目已经绑定当前批次。
- `enforce_v663_app_user_account_kind_change()`：账号已参与试点后，禁止改为与试点模式冲突的类别，避免历史证据被重新解释。
- 三个触发器函数仅允许 `service_role` 执行；迁移结束前运行全库安全审计并在发现问题时失败回滚。

管理员可在安全中心明确设置账号类别；当前登录管理员必须保持真实用户属性。类别变更经过服务端校验、操作审计和数据库历史冲突触发器三层门禁。

## 4. 接口与报告

`GET /api/operations-center/pilot-acceptance` 新增 `data.preflight`，返回每项检查的状态、数量、说明和操作入口。四角色匹配使用最大二分匹配计算，不能用同一个用户重复满足两个角色。

飞书候选查询强制 `project_id is not null`，并再次验证项目属于当前组织、数据空间和当前试点范围。接口只做前置筛选，最终写入仍由数据库触发器把关。

Markdown 验收报告新增：

- 五个项目的代码、名称和数据空间；
- 四角色账号类型、签署状态和签署时间；
- 16模块结论及安全化证据摘要；
- 黄金链 A/E 状态；
- 飞书三类动作、尝试次数、成功与恢复状态；
- 最近追加式试点事件。

证据中的敏感正文、令牌、密钥和个人配置不会原样写入下载报告。

## 5. 验收边界

自动化测试必须覆盖五项目与四人角色匹配、数据库身份触发器、飞书项目范围触发器、接口候选过滤和页面操作入口。生产迁移完成后，还需验证迁移历史、账号分类、唯一索引、三个触发器、客户端权限与全库安全审计。

V6.6.3 仍是技术验收候选。正式进入 V7.0 前，必须由五个真实项目、四位不同真实人员、两条黄金链、三类真实飞书写入和至少一次真实失败恢复共同形成证据，系统不会自动代签或伪造结果。
