# Product Life Journey

## 2026-07-16 - V6.6.7：数据分类目标镜像自动闭环

- **版本：** 6.6.7
- **用户变化：** 用户完成高风险确认后，不再需要返回集成中心手工触发全量对账；系统自动对刚写入的记录完成目标空间镜像。
- **事实源门禁：** 个人连接与组织共享连接必须指向同一Base、领域表和同步流水表，防止写入错误台账。
- **数据安全：** 定向对账只读取一个记录并使用非完整快照，不影响同表其他记录；失败可安全重试且不重复改写飞书。
- **数据库变化：** 无新增migration，复用V6.2既有事务RPC和48份已应用migration。
- **正式门禁：** 237条隔离记录仍未被自动分类，真实飞书回执、五项目、四真人和黄金链仍需业务人员完成。

## 2026-07-16 - V6.6.6：飞书数据分类受控写回闭环

- **版本：** 6.6.6
- **用户变化：** PMO可在隔离治理台直接选择分类、填写依据并创建飞书写回确认；已有草稿在刷新后继续显示状态和确认入口。
- **外部写入边界：** 页面和治理API不直接改写飞书；个人飞书、二次确认、风险确认和同步流水缺一不可。
- **安全变化：** 样例/测试标记禁止晋升正式；未重新对账前不创建正式项目，也不为隔离记录虚构项目ID。
- **数据库变化：** 新增分类草稿表、确认关联、4个事务函数及状态/租约/幂等门禁；第48份migration已应用且安全审计0。
- **正式门禁：** 功能降低了数据负责人操作成本，但真人业务分类、5个正式项目、4位真人、黄金链和CEO签署仍不能自动代办。

## 2026-07-16 - V6.6.5：生产分类口径一致性补丁

- **版本：** 6.6.5
- **用户变化：** README、项目状态、任务日志与产品生命旅程统一展示线上治理API的230条样例建议、7条人工判断和0条正式候选。
- **口径边界：** 原始载荷启发式扫描的234条“含字样”不能替代白名单分类器的最终治理结论。
- **系统变化：** 不改变治理页面、分类算法、权限、接口或数据，不执行新增SQL，不修改飞书源表。
- **正式门禁：** 5个真实项目、4位不同真人、黄金链A/E、飞书三类回执和失败恢复仍必须由真实业务完成。

## 2026-07-16 - V6.6.4：飞书隔离分类治理与正式试点启动包

- **版本：** 6.6.4
- **用户变化：** PMO 不再只看到“237条隔离”，可以逐条查看分类建议、依据和飞书补值动作，并下载中文治理 CSV。
- **数据变化：** 只有飞书明确标注“正式”的记录可成为 production 候选；样例、测试和未知数据不会被自动提升。
- **试点变化：** 一键导出正式试点启动包，集中列出五项目、四真人、黄金链、飞书回执和失败恢复缺口。
- **生产事实：** 原始载荷启发式扫描曾检出234条带样例/测试字样；生产治理API最终输出为230条样例建议、7条人工判断、0条正式候选，这批数据不能替代真实项目试点。
- **数据库：** 无新增迁移，不删除、不重分类既有隔离记录，等待飞书源端由业务负责人补齐中文字段后幂等重跑。
- **边界：** 系统不代签、不伪造 production、不自动执行外部写入；真人签署和真实回执仍是正式门禁。

## 2026-07-16 - V6.6.3：正式试点身份与证据门禁

- **版本：** 6.6.3
- **用户变化：** 正式试点页面在开始前显示五项目、四位不同真实人员、黄金链 A/E 和三类飞书写入的准备状态及操作入口。
- **身份变化：** 账号类型由数据库维护；真实用户、测试账号和服务账号不能跨试点模式混用。
- **证据变化：** 无稳定项目或不属于当前批次的飞书成功记录不能进入试点证据，历史记录仍完整保留。
- **报告变化：** 下载报告包含完整项目、角色签署、模块、黄金链、飞书尝试与事件审计清单，同时隐藏敏感证据正文。
- **边界：** 自动化只证明门禁有效，不代替四位真实人员本人签署、真实业务数据或真实飞书失败恢复。

## 2026-07-16 - V6.6.2：四角色汇报关系职责分离门禁

- **版本：** 6.6.2
- **角色变化：** PM、运营、PMO、CEO 的活动汇报链必须由不同用户承担，同一用户不能跨角色向自己汇报或升级。

- **界面变化：** 管理员建立汇报关系时，接收人不再包含当前上报人，身份冲突时无法提交。
- **数据库变化：** 新增 `20260716004254_v662_reporting_separation_guard.sql` 和已验证约束；活动自循环为 0，安全审计为 0。
- **边界：** 四个测试账号只能证明技术职责分离，正式试点仍必须由四位不同真实人员本人完成。

## 2026-05-14T13:45:00+08:00 - 新增功能：项目全流程交付管理蓝图 V3

### 任务元数据
- **指令类型**：新增
- **时间戳**：2026-05-14T13:45:00+08:00
- **关联版本号**：v3.0

### 逻辑链路
- **AI 思考过程**：用户提供了蓝图业务模块的文本结构，要求完美复刻图片样式并确保可修改。分析需求后，决定采用水平泳道图（Swimlane Flowchart）的设计模式来展示业务场景，使用 TailwindCSS 构建精美的卡片和连线效果。同时为了满足可修改的要求，引入了双击切换 `contentEditable/input` 的 React State 内联编辑模式，满足用户在不用第三方库的条件下直接修改文字的需求。
- **选用的关键调优解决方案**：使用 React 的 `useState` 结合内联 `input` 构建节点内容的双击编辑功能；设计专门的“贯穿全流程”模块横跨项目管理层级。

### 执行细节
- **调用的具体工具**：`list_dir`, `view_file` (分析已有页面), `write_to_file` (创建 `src/app/blueprint-v3/page.tsx`), `replace_file_content` (更新首页路由)。
- **遇到的技术阻碍**：由于无法直接查看用户的图片附件，采用了标准的 UI 最佳实践实现了用户文字描述的泳道结构，并在页面顶端预留了带容错提示的 `<img>` 占位，指导用户放置附件原图。
- **最终处理方案**：成功创建 `blueprint-v3` 页面并嵌入首页。

### 未来演进
- 考虑引入图形化连线库（如 React Flow）来实现更自由的拖拽编辑。
- 增加将修改后的蓝图导出为图片或 PDF 的功能。

### 阶段总结
完成了 V3 蓝图视图的开发，包含了图片附件展示区域与可交互的文本编辑流程图，与用户的具体字段需求完全对应。
